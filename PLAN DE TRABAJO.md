1. # **Metodología**
El desarrollo del proyecto seguirá la **metodología del Ciclo de Vida Clásico en Cascada**, que contempla las fases de:

0. **Definición de requisitos** (semana 1): levantar la información de qué se necesita (entrevistas, observación, encuestas, etc.).
0. **Análisis de requisitos** (Semanas 2): estudiar esa información para ver si es viable, cómo se va a cumplir y qué implicancias tiene.
0. **Diseño** (semanas 3–4).
0. **Desarrollo** (semanas 5–8).
0. **Pruebas y validación** (semanas 9–10).
0. **Implementación** (semanas 11–12).
0. **Documentación** (semanas 13–14).
0. **Cierre y mantenimiento inicial** (semana 15).

Cada fase generará un entregable (documentos, prototipos, código, informes, manuales) que servirá de evidencia para la universidad y la institución.

*Con base en la literatura especializada (Pressman, Sommerville, IEEE/ISO 12207), lo correcto es:*

0. ***El modelo en cascada es una metodología de desarrollo de software**.*

   *Se considera el primer modelo formal de ciclo de vida de software (propuesto por Winston Royce en 1270).*

0. *Es **metodología** porque define un proceso rígido y secuencial: requisitos → diseño → implementación → pruebas → mantenimiento. Cada fase debe*

   *completarse antes de pasar a la siguiente.*

0. *No se clasifica como marco de trabajo (framework) ni como enfoque ágil.*
0. *Está dentro de la familia de **modelos de ciclo de vida de software** y se caracteriza por ser lineal y predictivo.*
1. # **Justificación de la metodología**
Se eligió el **modelo en cascada** como metodología de desarrollo debido a que se adapta de manera adecuada a la naturaleza de las prácticas preprofesionales. Si bien se reconoce que es el modelo más antiguo (propuesto por Winston Royce en 1970), aún se emplea en contextos académicos y de proyectos formativos porque:

1. **Claridad y estructura:** define fases bien delimitadas (definición de requisitos, análisis, diseño, construcción, pruebas, implementación, documentación y cierre), lo que permite organizar de forma lineal las actividades del plan de trabajo.
1. **Facilidad de control y seguimiento:** al ser un modelo secuencial, es sencillo alinear cada fase con las semanas del cronograma de prácticas, facilitando el control de avances y la evaluación por parte del supervisor y del docente.
1. **Adecuación al contexto institucional:** el proyecto no busca un producto con cambios iterativos constantes como en un entorno ágil, sino una implementación con entregables progresivos y verificables, que culmina en la entrega final del sistema.
1. **Nivel de complejidad del proyecto:** si bien el proyecto no es sumamente complejo, sí es **concreto y específico**, lo que hace viable un desarrollo ordenado siguiendo un camino mayormente lineal, sin requerir muchas iteraciones.
1. **Respaldo académico y normativo:** según la literatura especializada (Pressman, Sommerville, IEEE/ISO 12207), el modelo en cascada sigue siendo una referencia válida en la enseñanza de la ingeniería de software por su valor didáctico y por sentar las bases de la planificación ordenada.

En conclusión, aunque existen metodologías más modernas como las ágiles o iterativas, el **modelo en cascada** resulta pertinente en este contexto porque asegura una ruta clara de trabajo, es coherente con el cronograma de las prácticas, y facilita tanto la documentación como la evaluación del proyecto por la universidad y la institución receptora.

**VIII. Anexos de herramientas, tecnologías utilizadas**

**A. Conozco / usaré “tal cual”**

- **Frontend React (PWA) – uso tal cual**\
  Para qué se usará: construir la interfaz web instalable y responsiva de la agenda.

  **En qué consiste:** aplicación React con capacidades PWA (manifest y service worker) para caché, carga rápida y modo offline básico.

- **Base de Datos en Supabase (PostgreSQL + Auth)**\
  Para qué se usará: persistir incidencias, usuarios/roles y auditoría; manejar autenticación.\
  **En qué consiste:** Postgres gestionado con APIs automáticas, reglas RLS para acceso por roles y módulo de autenticación integrado.
- **CI/CD con GitHub (repos, ramas y workflows)**\
  Para qué se usará: versionar el código y automatizar pruebas/despliegues.\
  **En qué consiste:** repositorio Git con ramas main/dev y workflows de GitHub Actions para build y deploy.
- **Figma**
- **Bizagi Modeler**

**B. Por aprender / aplicaré durante la práctica**

- **Cloudflare Pages — despliegue del frontend**\
  Para qué se usará: alojar el PWA en una CDN global con previews por rama.\
  **En qué consiste:** servicio de hosting estático que compila y publica el sitio desde el repo.
- **Cloudflare Workers — backend serverless (Wrangler)\
  Para qué se usará:** exponer APIs ligeras (Edge) para notificaciones y acceso a datos.\
  **En qué consiste:** funciones serverless ejecutadas en la red de Cloudflare; se gestionan con la CLI Wrangler.
- **Web Push API con VAPID — suscripción, envío y recepción\
  Para qué se usará:** enviar notificaciones Push a docentes/padres desde el navegador.\
  **En qué consiste:** claves VAPID para firmar mensajes, suscripción del navegador, envío desde el backend y recepción en el service worker.
- **Seguridad, datos personales y cumplimiento\
  Para qué se usará:** proteger información sensible y cumplir con principios de privacidad.\
  **En qué consiste:** HTTPS, CORS controlado, roles y RLS en BD, minimización de datos, políticas de retención y textos de consentimiento/privacidad.

**C. Elementos transversales de este anexo**

- **Stack y versiones mínimas\
  React ≥ 18; Node ≥ 20 (si aplica); NPM/PNPM; Supabase Postgres 15; Cloudflare Wrangler ≥ 3; GitHub Actions runners Ubuntu-latest; Web Push API (VAPID).**
- **Arquitectura lógica de la solución (PWA + Edge + BD)**
- **PWA (React) ↔ Workers (Edge API) ↔ Supabase (Postgres/Auth).\
  Flujo de notificaciones:** Servidor (Worker) → Push Service → PWA (Service Worker).
